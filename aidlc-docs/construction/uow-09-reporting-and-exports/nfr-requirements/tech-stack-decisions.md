# Tech Stack Decisions - UOW-09 Reporting and Exports

## Decision Summary

| Area | Decision | Status |
|---|---|---|
| Language | TypeScript | Approved |
| Package | `@spa-bridge/core-reporting` | Approved |
| Canonical format | Versioned JSON object model | Approved |
| Human-readable export | Deterministic Markdown renderer | Approved |
| HTML export | Escaped sanitized HTML renderer | Approved |
| Runtime dependencies | Prefer none beyond workspace packages | Approved |
| Testing | Vitest plus fast-check | Approved |
| Hashing | Node.js crypto stable content hashes | Approved |

## Rendering Stack

UOW-09 will use TypeScript-native rendering:

- JSON is produced from the canonical report model.
- Markdown is generated by deterministic string rendering helpers.
- HTML is generated by deterministic escaped string rendering helpers.
- Raw HTML snippets from inputs are not accepted.
- Browser DOM rendering is not required for this unit.

## Dependency Policy

The reporting package should start with no new runtime dependencies beyond existing workspace packages.

Allowed dependency sources:

- `@spa-bridge/core-model`
- existing workspace package contracts required for report input typing
- Node.js built-ins such as `node:crypto` for stable hashes

Conditional dependency path:

- A sanitizer or structured serializer may be added only if manual escaped rendering proves insufficient.
- Any added dependency must use an exact version.
- Any added dependency must be covered by the workspace lockfile and dependency audit instructions.

Rejected options:

- Broad Markdown/HTML rendering libraries by default.
- Browser DOM rendering for report generation.
- Remote or SaaS report rendering.
- Template engines that accept raw HTML from report inputs.

## Schema Strategy

The canonical report model must be versioned:

- `schemaVersion` belongs in every canonical report.
- Renderer version belongs in export metadata.
- Additive changes are preferred.
- Breaking schema changes require an explicit schema version increment.
- CLI and Web UI consume the same schema.

## Deterministic Export Strategy

Reports must be reproducible for equivalent inputs:

- Stable section order.
- Stable item order.
- Stable report and item IDs.
- Caller-controlled timestamps.
- Stable content hashes.
- Byte-stable Markdown and HTML output.

This strategy supports snapshot tests, PBT replay, report comparisons, and stable review workflows.

## Security Strategy

Reporting owns its own validation layer even though UOW-05 handles masking and provider policy.

The reporting package must:

- reject unsafe raw source, prompt, provider response, and secret-like fields
- never unmask or restore sensitive values
- escape Markdown and HTML output
- represent paths as safe refs
- fail closed for unsafe content
- emit safe diagnostics only

## Testing Stack

| Tool | Purpose |
|---|---|
| Vitest | Example-based unit tests and package-level tests |
| fast-check | Property-based tests for report invariants |
| TypeScript compiler | Static validation and package build |
| Node.js crypto | Stable hash generation for report/export refs |

## Required PBT Generators

Code generation must provide reusable generators for:

- canonical reports
- report input bundles
- diagnostics
- manual review items
- AI-assisted decisions
- quality summaries
- traceability links
- safe source/generated refs
- export render options

## Build and Test Implications

The root workspace build and test scripts must include `@spa-bridge/core-reporting` after implementation.

Expected package commands:

```bash
npm run build --workspace @spa-bridge/core-reporting
npm run test --workspace @spa-bridge/core-reporting
```

## Decision Rationale

This stack keeps reporting small, deterministic, and safe. It also gives UOW-10 CLI and UOW-11 Web UI one shared report model instead of duplicated formatting and grouping behavior.
